# v2.0 — 客户级容器隔离架构

> **目标**：从"agent = Railway service"演进到"客户 = Railway service / 容器，
> agents = 容器内子路由"。强物理隔离，多 agent 共享客户数据，B2B 合规友好。
>
> **现状**：v1.4 一客户一 agent，撞巧 1:1 映射 service。
> **触发点**：第一个客户开始有第 2 个 agent（不论同客户加 agent，还是新客户进来）。
> **不要提前优化**：1-3 客户阶段，v1.4 现状就是隐式的客户隔离。

---

## 为什么不走 multi-tenant agent

### 选项对比

| | 客户=容器（v2.0 目标） | agent template 多租户 | 一客户一 service（v1.4 现状） |
|---|---|---|---|
| 跨客户数据泄露风险 | **0**（容器边界） | 中（漏个 `WHERE tenant_id` 就漏） | **0** |
| 容器内多 agent 共享数据 | ✅（同客户 sqlite/缓存） | N/A | ❌（每 agent 独立 DB） |
| 合规友好（金融/医疗类客户） | ✅ | ❌ | ✅ |
| 客户加新 agent 工作量 | 加 dir + tool 注册 | 同左 + schema 改 | 新 service 部署 |
| 月成本（10 客户） | $50-100 | $30-50 | $50-100 |
| 月成本（100 客户） | $300-500 | $80-150 | $1000+（不可行） |

**B2B 选 v2.0**：贵 2-3x 但**客户买完不担心数据混**，对企业销售好讲。

### LLM 幻觉/代码 bug 是真实风险

multi-tenant agent 模式下，每个 SQL 都依赖 `WHERE tenant_id=?` 的严格性。一旦：
- 某个查询忘加 `WHERE` 子句
- LLM 用 tool 时把另一个客户的 `tenant_id` 传进去
- 缓存 key 没 namespace 客户

→ 跨客户数据泄露。容器边界让这种 bug **不可能**跨界。

## 目标架构

### Railway 项目结构

```
yunwei-platform Railway project:

  platform-app                       (1) 多租户路由 + auth + 反代
  Postgres                           (1) 平台元数据(users, tenants, ACL, proxy_log)
  Redis                              (1) 共享 cache + nonce store

  customer-yinhu                     ← 1 容器 = 1 客户(强隔离)
    /app/agents/super-xiaochen/     ← CRM 销售 agent
    /app/agents/bid-analyzer/       ← (将来)合同分析 agent
    /app/agents/finance-bot/        ← (将来)财务对账 agent
    /data/yinhu.db                   ← 银湖**所有 agent 共享**一份 sqlite
    /data/yinhu-files/               ← (将来)文档存储
    env:  CUSTOMER_ID=yinhu, KINGDEE_*=<银湖凭证>, ...

  customer-abc                       ← 客户 B 独立容器
    /app/agents/sales-agent/
    env:  CUSTOMER_ID=abc, KINGDEE_*=<B 凭证>

  customer-def                       ← 客户 C
    ...
```

### 平台反代映射

```
Browser GET /yinhu/super-xiaochen/chat
         │
         ▼
platform-app catch_all
         │
         │ tenants table 查
         │   client_id=yinhu, agent_id=super-xiaochen
         │   → container_url='http://customer-yinhu.railway.internal:8000/super-xiaochen'
         │
         ▼  HMAC-sign + 反代
http://customer-yinhu.railway.internal:8000/super-xiaochen/chat
         │
         ▼
customer-yinhu container
   FastAPI dispatcher: APIRouter(prefix='/super-xiaochen').include_router(super_xiaochen_app)
                       APIRouter(prefix='/bid-analyzer').include_router(bid_analyzer_app)
   → super_xiaochen_app handles /chat
```

### 容器内多 agent 共享什么 / 隔离什么

| 资源 | 共享/隔离 |
|---|---|
| sqlite `/data/yinhu.db` | **共享**（客户级数据：客户档案、订单、应收等） |
| LLM tools (TOOLS array) | 每 agent 独立注册 |
| MCP server / Kingdee client | 实例化时共享（同 KINGDEE_* env） |
| chat_turns / user_memories table | 共享但带 `agent_id` 字段区分 |
| onboarding profile | 共享（同客户员工 profile 跨 agent） |
| HMAC nonce store | 全局 Redis（不在容器内） |

## 触发条件 / 何时开始

不是日历驱动，是**业务事件**驱动：

| 触发事件 | 行动 |
|---|---|
| 第 1-3 客户都只 1 个 agent | 不动，v1.4 维持 |
| 任意客户加第 2 个 agent | 启动 v2.0 阶段 1（重构那 1 个客户） |
| 同时有 5+ 客户 | 启动 ops 自动化（`add-customer.sh`） |
| 20+ 客户 / 单 service 撑不住 | v3.0 上 K8s |

**不要提前优化**——许总现在 1 个客户 1 个 agent，v1.4 隐式就是客户隔离。

## 演进步骤

### 阶段 1：单客户 multi-agent runtime（trigger：客户加第 2 个 agent）

**目标**：把 yinhu 容器从"单 agent"变成"多 agent dispatch"。

**代码改动**（yinhu-super-xiaochen repo → rename `customer-yinhu`）：

1. **目录重组**：
   ```
   customer-yinhu/
     ├── runtime/                    # 共享:auth, db, kingdee_client
     │   ├── auth.py
     │   ├── db.py
     │   └── kingdee_client.py
     ├── agents/
     │   ├── super_xiaochen/         # 之前的 web_agent.py + test_agent.py 内容
     │   │   ├── tools.py            # TOOLS array
     │   │   ├── dispatcher.py       # _dispatch
     │   │   └── routes.py           # FastAPI APIRouter
     │   └── (future agents...)
     └── main.py                     # FastAPI app, mount each agent's APIRouter
   ```

2. **`main.py` 入口**：
   ```python
   from fastapi import FastAPI
   from agents.super_xiaochen.routes import router as super_xiaochen_router
   # from agents.bid_analyzer.routes import router as bid_analyzer_router

   app = FastAPI(lifespan=_lifespan)  # 共享 lifespan: periodic sync, etc.
   app.include_router(super_xiaochen_router, prefix="/super-xiaochen")
   # app.include_router(bid_analyzer_router, prefix="/bid-analyzer")
   ```

3. **`tenants.container_url` schema 调整**：
   - 旧：`http://agent-yinhu-super-xiaochen.railway.internal:8000`
   - 新：`http://customer-yinhu.railway.internal:8000/super-xiaochen`
   - 平台反代逻辑不变（`container_url + subpath`）

4. **HMAC 验签** 加 `agent_id` 到 payload（已经在了，确认）

5. **`agent_id` 路由参数** 传到所有 tool 调用，DB 写入时带：
   ```python
   await db.execute(
       "INSERT INTO chat_turns (agent_id, user_id, ...) VALUES (?, ?, ...)",
       (agent_id, user_id, ...),
   )
   ```

6. **schema migration**：
   ```sql
   ALTER TABLE chat_turns ADD COLUMN agent_id TEXT NOT NULL DEFAULT 'super-xiaochen';
   ALTER TABLE user_memories ADD COLUMN agent_id TEXT NOT NULL DEFAULT 'super-xiaochen';
   -- 等所有客户级表都加 agent_id 字段
   CREATE INDEX idx_chat_turns_agent ON chat_turns(agent_id, user_id);
   ```

**Railway 调整**：
- service rename `agent-yinhu-super-xiaochen` → `customer-yinhu`
- env vars 不变（KINGDEE_* 等仍是 yinhu 客户的）
- Volume 不变（同一份 sqlite）

**预计工作量**：3-5 天

### 阶段 2：自动化加客户（trigger：第 2 个客户进来）

**目标**：写脚本一键加新客户容器。

**`ops/add-customer.sh`**：
```bash
#!/usr/bin/env bash
# Usage: ./add-customer.sh <client_id> <customer-creds.env>
#
# Creates a Railway service for a new customer:
# - new container based on customer-template image
# - per-customer env vars (KINGDEE_*, ANTHROPIC_API_KEY, etc.)
# - new Volume mounted at /data
# - DNS not touched (platform routes via tenants.container_url)
# - inserts into platform Postgres tenants + grants ACL

CLIENT_ID="$1"
CREDS_FILE="$2"

# 1. Railway: create service from template repo
railway api 'mutation { serviceCreate(input: {
  projectId: "...",
  name: "customer-'"$CLIENT_ID"'",
  source: { repo: "Fiveoranges-AI/customer-template" }
}) { id } }'

# 2. Set env vars from CREDS_FILE
while IFS='=' read -r key val; do
  railway variables set "$key=$val" --service customer-$CLIENT_ID
done < "$CREDS_FILE"

# 3. Create Volume at /data
railway api 'mutation { volumeCreate(input: {
  serviceId: "...",
  mountPath: "/data"
}) }'

# 4. Wait for first deploy
railway api 'mutation { serviceInstanceDeployV2(...) }'

# 5. Platform: add tenant rows for each agent the customer is licensed to
for AGENT in super-xiaochen bid-analyzer; do
  railway run --service platform-app -- \
    python -m platform_app.admin add-tenant "$CLIENT_ID" "$AGENT" \
    --display-name "$CLIENT_ID 的 $AGENT" \
    --container-url "http://customer-$CLIENT_ID.railway.internal:8000/$AGENT"
done
```

加新客户从"半天 dashboard 操作"降到"`./add-customer.sh abc abc.env`"。

**预计工作量**：1 天

### 阶段 3：customer-template repo 抽离（trigger：3+ 客户）

**目标**：把"customer 容器"代码抽到独立 repo，多客户共享同一份。

```
GitHub:
  Fiveoranges-AI/customer-template     ← 1 份代码,所有客户用
    ├── runtime/                       # 共享 auth/db/kingdee_client
    ├── agents/                        # 所有 agent 实现(每个 agent 一个 dir)
    │   ├── super-xiaochen/
    │   ├── bid-analyzer/
    │   ├── finance-bot/
    │   └── ...
    ├── main.py                        # 启动时按 ENABLED_AGENTS env 注册
    └── Dockerfile

  Fiveoranges-AI/customer-yinhu        ← 银湖私有 fork(如果有自定义 agent)
  Fiveoranges-AI/customer-abc          ← 客户 B 私有 fork
```

每客户 Railway service 指向 `customer-template` repo（同 image），靠 env `ENABLED_AGENTS=super-xiaochen,bid-analyzer` 启用对应 agent。

`main.py` 启动时：
```python
enabled = os.environ.get("ENABLED_AGENTS", "").split(",")
for agent_id in enabled:
    module = importlib.import_module(f"agents.{agent_id.replace('-', '_')}.routes")
    app.include_router(module.router, prefix=f"/{agent_id}")
```

**好处**：所有客户跑同一个 image，只要 `customer-template` 修 bug 全客户自动获得。客户专属代码用 fork。

**预计工作量**：1 周

### 阶段 4：K8s 上云（trigger：20+ 客户 OR Railway 撑不住）

离开 Railway，上 K8s（GKE / EKS / ACK）：

```yaml
# customer-yinhu Namespace
apiVersion: v1
kind: Namespace
metadata: { name: customer-yinhu }
---
apiVersion: apps/v1
kind: Deployment
metadata: { name: customer-yinhu, namespace: customer-yinhu }
spec:
  replicas: 1
  template:
    spec:
      containers:
      - name: customer
        image: ghcr.io/fiveoranges-ai/customer-template:latest
        env: [...]
        resources:
          requests: { memory: "256Mi", cpu: "100m" }
          limits:   { memory: "1Gi",   cpu: "1" }
```

加客户 = 创建 namespace + deployment（用 Helm chart 或 Operator）。

**好处**：autoscale per customer，更便宜（multi-customer 共享 node pool），细粒度 resource limits。

**坏处**：K8s 复杂度高，要 SRE 时间。

**预计工作量**：2-4 周（不含 SRE 学习曲线）

## 跨阶段一致的设计原则

1. **客户数据在客户容器内** —— 同 sqlite/Volume，跨容器物理不可触达
2. **平台元数据在 platform Postgres** —— users / tenants / ACL / proxy_log，跨客户可查（管理面用）
3. **agent_id 是容器内路由 + DB 字段** —— 同客户内多 agent 区分
4. **container_url 是 (client_id, agent_id) 的复合 key** —— 可同 host 不同路径
5. **HMAC payload 含 client + agent** —— 防止 agent A 的请求被路由到 agent B 时验签通过

## 安全 boundaries 总结

```
┌─ Browser ────────────────────────────────────────────┐
│  Cookies: app_session, app_csrf (per platform user)  │
└────────────────────┬─────────────────────────────────┘
                     │ HTTPS, SameSite=Lax
┌────────────────────▼─────────────────────────────────┐
│  platform-app (multi-tenant)                          │
│   - reads cookie → user_id                            │
│   - reads url → (client_id, agent_id)                │
│   - checks ACL: user_tenant table                     │
│   - signs HMAC with (client, agent, user) headers    │
└────────────────────┬─────────────────────────────────┘
                     │ Railway internal network
                     │ (HMAC-signed reverse proxy)
┌────────────────────▼─────────────────────────────────┐
│  customer-{client_id} container                       │
│   - verifies HMAC signature                           │
│   - dispatches to /agent_id/* router                  │
│   - tools query /data/{client}.db (容器局部)          │
│   - calls Kingdee / external APIs (per-customer creds)│
└──────────────────────────────────────────────────────┘
```

跨客户**任何方向**都打不通（平台反代靠 HMAC + ACL，agent 容器没暴露公网）。
