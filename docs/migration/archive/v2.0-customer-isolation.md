# v2.0 — Runtime / Data Plane / Security Architecture

> Status: design baseline
> Principle: **Day 1 按最终边界设计；Day 1 不一定按最终基础设施部署。**

本文件替代旧版“客户级容器隔离架构”。旧版方向仍然成立，但过于单一地把
v2 写成“每个客户一个容器”。结合智通客户的 Free Trial / Lite / Pro /
Max 产品分层，v2 应该表达为：

- Platform 是共享控制平面。
- Runtime 是按套餐和风险分层的执行平面。
- Data Plane 是按套餐和数据敏感度分层的数据平面。
- Agent 是 runtime 内的业务模块，不是公司隔离边界。
- Local app / desktop app 只作为可选 connector，不作为默认主 runtime。

最终目标不是一开始就上最重的 K8s / multi-region / service mesh，而是从
第一天固定正确边界，让未来从 Railway 迁到 K8s、从 pooled runtime 迁到
dedicated runtime、从 pooled data store 迁到 private data store 时，不推翻
业务代码和产品接口。

---

## 1. Product Constraints

智通客户采用 AI-native SaaS land-and-expand 路径：

| Plan | Product goal | Data / runtime implication |
|---|---|---|
| Free Trial | 让客户快速体验 AI 整理资料的 wow moment | 低成本 pooled runtime，严格额度和保留期，不接 ERP |
| Lite | 不接系统的 AI 客户资料整理工具 | pooled Lite data/runtime，可长期保存但仍以逻辑隔离为主 |
| Pro | 接金蝶 / Excel / 钉钉的 AI 经营助手 | 每公司 dedicated customer runtime，至少 dedicated schema |
| Max | AI SaaS + 人工实施 + 数据治理 + 私有化/专属服务 | dedicated/private runtime 和 data plane，可客户云/本地 connector |

这意味着 v2 不能只有一种 runtime 形态：

- Free/Lite 如果每家公司都起专属 runtime，获客成本会过高。
- Pro/Max 如果仍共用一个 multi-tenant harness，ERP 凭据、经营数据、老板日报、
  回款库存等敏感场景的安全和定制风险会过高。

正确边界是：**低价层共享资源换低成本，高价值层独立资源换安全、定制和高客单价。**

## 2. Core System Boundary

```
Browser / Mobile Web / Optional Desktop Connector
        |
        v
Shared Platform Control Plane
  - Auth / users / enterprises / members
  - plan / entitlement / quota
  - tenant registry / runtime binding
  - ACL / policy metadata
  - HMAC gateway / reverse proxy
  - audit / usage / deployment control
        |
        v
Execution Plane
  - pooled-trial runtime
  - pooled-lite runtime
  - dedicated customer runtime for Pro
  - private / hybrid runtime for Max
        |
        v
Data Plane
  - pooled trial/lite data stores
  - dedicated Pro schema/database
  - Max dedicated project / customer cloud / local connector
```

Platform 只 owns 控制平面。客户业务数据不应该长期混进 platform metadata DB。
Platform 可以保存 metadata、routing、quota、audit、usage、job state，但不应该
成为所有客户业务主数据的唯一大库。

## 3. Execution Plane

### 3.1 Runtime Modes

| Mode | Used by | Isolation | Cost | Notes |
|---|---|---|---|---|
| `pooled_trial` | Free Trial | logical tenant isolation | lowest | short TTL, strict quota, no ERP |
| `pooled_lite` | Lite | logical tenant isolation + RLS | low | upload-only customer profile |
| `dedicated_customer` | Pro | one runtime per company | medium | ERP / DingTalk / Excel secrets live here |
| `private_hybrid` | Max | dedicated/private/customer-owned | high | customer cloud, VPC, local connector |

### 3.2 Runtime Registry

Platform should route through a registry, not through hardcoded deploy knowledge.

Conceptual table:

```text
runtimes
  runtime_id
  mode                  -- pooled_trial | pooled_lite | dedicated_customer | private_hybrid
  provider              -- railway | k8s | customer_cloud | local_connector
  endpoint_url          -- opaque base URL
  health
  version
  created_at

tenant_runtime_bindings
  enterprise_id
  agent_id
  runtime_id
  route_prefix          -- e.g. /super-xiaochen
  container_url         -- opaque endpoint used by platform proxy
  active
```

`container_url` remains an opaque endpoint. Today it may be:

```text
http://customer-yinhu.railway.internal:8000/super-xiaochen
```

Later it may become:

```text
http://customer-yinhu.customer-yinhu.svc.cluster.local:8000/super-xiaochen
```

Platform proxy should not care. It only does:

```text
(client_id, agent_id) -> container_url -> HMAC-sign + reverse proxy
```

### 3.3 Dedicated Customer Runtime

For Pro and above:

```text
customer-yinhu
  /app/runtime/
    auth.py
    db.py
    secrets.py
    tool_registry.py
    policy.py
  /app/agents/
    super_xiaochen/
    daily_report/
    finance_bot/
  /data/...
  env:
    CUSTOMER_ID=yinhu
    ENABLED_AGENTS=super-xiaochen,daily-report
    KINGDEE_*=...
    DINGTALK_*=...
```

One company can have many agents. Agents are mounted as subroutes inside the
customer runtime:

```python
enabled = os.environ.get("ENABLED_AGENTS", "").split(",")
for agent_id in enabled:
    module = importlib.import_module(f"agents.{agent_id.replace('-', '_')}.routes")
    app.include_router(module.router, prefix=f"/{agent_id}")
```

### 3.4 Local App / Connector

Desktop/mobile/local app should not be the default harness. It is a connector:

- access local ERP client / local files / LAN-only systems
- run outbound-only tunnel or mTLS connection
- sync data or serve controlled tool calls

Default product runtime stays cloud-side. Max can add local connector when data
residency or local system access requires it.

## 4. Data Plane

### 4.1 Data Categories

Do not mix these into one undifferentiated database:

| Category | Examples | Owner |
|---|---|---|
| Platform metadata | users, enterprises, plans, sessions, tenants, ACL, billing | Platform DB |
| Usage / quota | upload count, question count, trial state, limits | Platform DB |
| Raw customer data | PDFs, Excel, business cards, screenshots, OCR JSON | Data plane storage |
| Structured business data | customers, contacts, contracts, orders, receivables, inventory | Data plane DB |
| AI-ready data | bronze/silver/gold, embeddings, memory, daily-report inputs | Data plane DB/storage |
| Audit | data access, tool calls, exports, permission changes | Platform + data plane |

### 4.2 Data Plane Modes

| Mode | Used by | Storage model | Notes |
|---|---|---|---|
| `pooled_trial` | Free Trial | shared tables with `enterprise_id`, shared object prefix | 7-30 day retention, auto purge |
| `pooled_lite` | Lite | shared Postgres + RLS, object prefix per tenant | low-cost paid tier |
| `dedicated_schema` | Pro default | shared Postgres cluster, schema per company | good balance of isolation and ops |
| `dedicated_db` | Pro high-risk / Max | database per company | stronger isolation, higher ops cost |
| `external_private` | Max | customer-owned DB/storage or private project | platform stores metadata only |

Recommended path:

```text
Free Trial -> pooled_trial
Lite       -> pooled_lite
Pro        -> dedicated_schema by default
Max        -> dedicated_db / external_private
```

This keeps Free/Lite acquisition cost low while giving Pro/Max the isolation
customers expect when connecting ERP and operating data.

### 4.3 Data Plane Registry

Data Center API should hide whether a tenant is pooled, schema-isolated,
database-isolated, or external.

Conceptual table:

```text
enterprise_data_planes
  enterprise_id
  plan
  data_mode             -- pooled_trial | pooled_lite | dedicated_schema | dedicated_db | external_private
  database_ref          -- secret manager ref, not raw URL in UI code
  schema_name
  storage_bucket
  storage_prefix
  runtime_id
  retention_policy
  created_at
```

Repository routing:

```python
data_plane = registry.get_data_plane(enterprise_id)

if data_plane.mode in ("pooled_trial", "pooled_lite"):
    repo = PooledRepo(shared_db, enterprise_id)
elif data_plane.mode == "dedicated_schema":
    repo = SchemaRepo(shared_db, schema=data_plane.schema_name)
elif data_plane.mode == "dedicated_db":
    repo = DedicatedRepo(secret_manager.get(data_plane.database_ref))
elif data_plane.mode == "external_private":
    repo = RemoteConnectorRepo(data_plane.connector_endpoint)
```

### 4.4 Data Center API

Frontend and agent tools should call stable APIs:

```text
GET  /api/data/health?enterprise=<id>
GET  /api/data/tables/customers/rows?enterprise=<id>
POST /api/data/upload
POST /api/data/manual/customers
POST /api/data/sync/kingdee
POST /api/data/snapshot/<enterprise_id>
```

The API routes internally to the correct data plane.

### 4.5 Bronze / Silver / Gold

Use a lakehouse-style contract, but avoid making local files the only online
source of truth.

```text
Bronze = raw landed data
  - uploaded files
  - OCR output
  - ERP raw snapshots
  - original source metadata

Silver = canonical AI-ready business tables
  - customers
  - contacts
  - contracts
  - orders
  - receivables
  - inventory

Gold = product outputs
  - CEO daily reports
  - risk alerts
  - dashboards
  - operating KPIs
```

Recommended storage split:

```text
Raw files          -> object storage
Bronze metadata    -> platform/data-plane DB
Silver online data -> Postgres schema/database
DuckDB/Parquet     -> snapshot/export/evaluator cache, not sole online DB
Embeddings/memory  -> tenant-scoped vector collection/table
```

### 4.6 Supabase Positioning

Supabase can be used as a component, not as the main agent runtime:

Good fit:

- Free/Lite Postgres + RLS
- object storage
- queues / cron for light jobs
- dedicated Max project when a customer wants a managed isolated project

Poor fit:

- FastAPI HMAC reverse proxy
- long SSE gateway
- heavy OCR / Excel / PDF workers
- customer agent harness runtime

The v2 architecture should allow Supabase as one data plane provider, while
keeping the execution plane container-based.

## 5. Security And Privacy

Security has two separate goals:

```text
Cross-enterprise isolation:
  Company A cannot access Company B's data.

In-enterprise authorization:
  Within one company, boss / finance / sales / assistant see different data.
```

### 5.1 Cross-Enterprise Isolation

| Plan | Isolation |
|---|---|
| Free Trial | pooled runtime/data, strict `enterprise_id`, quota, TTL, RLS |
| Lite | pooled Lite data with RLS and storage prefix policy |
| Pro | dedicated customer runtime, dedicated schema or DB, per-company secrets |
| Max | dedicated/private runtime and data plane, optional local connector |

For Pro and Max, sales/security language should be explicit:

> 贵公司的业务数据运行在独立 customer runtime 和独立数据空间中。平台只保存账号、
> 权限、路由、套餐、审计等元数据。

### 5.2 In-Enterprise Authorization

Do not rely on a single company-level permission. Need RBAC + ABAC + row-level
ownership.

Suggested roles:

| Role | Typical access |
|---|---|
| Owner / boss | all company data, all agents, user management, data connections, billing |
| Admin | most operations, member management, agent config |
| Finance | receivables, collections, invoices, financial risk |
| Sales | own customers, own orders, own follow-ups |
| Assistant | assigned customers/files/tasks, limited financial visibility |
| Viewer | reports / dashboards / read-only access |

Policy must support:

- role
- department / team
- owner_user_id / manager relationship
- resource type
- action
- field sensitivity
- row visibility

Conceptual platform tables:

```text
enterprise_members
  enterprise_id
  user_id
  role
  status

enterprise_user_profiles
  enterprise_id
  user_id
  department
  title
  manager_user_id

permission_groups
  enterprise_id
  group_id
  name

permission_group_members
  group_id
  user_id

data_policies
  enterprise_id
  policy_id
  resource_type       -- customer | order | receivable | file | report | agent | tool
  action              -- read | write | delete | export | connect
  condition_json
```

Business data should carry ownership/sensitivity fields where needed:

```text
customers
  enterprise_id
  owner_user_id
  team_id
  visibility
  sensitivity_level

orders
  enterprise_id
  customer_id
  owner_user_id
  department_id

receivables
  enterprise_id
  customer_id
  amount
  sensitivity_level
```

### 5.3 Agent Tool Authorization

UI hiding is not enough. Every agent tool must pass through a policy layer.

Tool call context:

```text
enterprise_id
user_id
agent_id
role
permission_context
request_id
```

Pattern:

```python
policy.require(user, action="read", resource="receivables")
scope = policy.row_scope(user, resource="receivables")
query = query.where(scope)
```

LLM must never construct direct SQL against tenant data. It can only call
approved tools/repositories that enforce policy.

Example:

- Owner asks: "哪些客户欠款最多？" -> full company receivables.
- Sales asks the same -> only customers owned by that sales user.
- Unauthorized user asks -> explain permission limit, do not leak aggregates.

### 5.4 Audit

Audit should cover at least:

```text
who
enterprise_id
action
resource_type
resource_id
agent_id
tool_name
timestamp
result_count
ip
request_id
status
```

Mandatory audited events:

- data export
- receivable / collection queries
- file upload / delete
- ERP / DingTalk connection changes
- agent tool data access
- permission changes
- plan upgrades and data-plane migrations

## 6. Platform Gateway Contract

The existing HMAC reverse proxy pattern remains the correct control boundary.

Request path:

```text
Browser GET /yinhu/super-xiaochen/chat
         |
         v
platform-app
  - reads session cookie
  - resolves user
  - checks enterprise/agent ACL
  - runs firewall/CSRF checks
  - resolves (client_id, agent_id) -> container_url
  - signs HMAC with client + agent + user + method + host + path + body hash
         |
         v
customer runtime
  - verifies HMAC
  - verifies tenant/client/agent
  - dispatches to /agent_id/* router
  - runs tool/data policy for each operation
```

`container_url` is the migration seam. Infrastructure can change behind it.

## 7. Upgrade And Migration Paths

### 7.1 Trial -> Lite

No runtime/data-plane migration required.

```text
1. update plan / entitlement
2. increase quota
3. increase retention
4. keep pooled data plane
```

### 7.2 Lite -> Pro

This is the important land-and-expand migration.

```text
1. provision dedicated customer runtime
2. provision dedicated schema or database
3. freeze writes briefly or enable dual-write window
4. export pooled tenant business rows and files
5. import into dedicated data plane
6. update enterprise_data_planes
7. update tenant_runtime_bindings / container_url
8. run smoke tests
9. keep old pooled data read-only for rollback window
```

Product trigger examples:

- user asks real-time receivable / payment status
- user asks order production or delivery status
- user asks inventory questions
- customer wants Kingdee / DingTalk / Excel sync
- customer needs boss daily report

### 7.3 Pro -> Max

```text
1. provision dedicated DB/project/VPC or customer-cloud deployment
2. replicate data
3. enable private/hybrid runtime
4. optionally deploy local connector
5. cut over data_plane registry and runtime binding
6. provide audit/export package
```

## 8. Deployment Evolution

### Phase 0 — Boundary-Compatible Current State

Do now:

- keep Platform as shared control plane
- keep `container_url` opaque
- keep HMAC reverse proxy boundary
- add runtime/data-plane registry concepts
- fix deployment config drift:
  - Dockerfile build context should match compose/Railway
  - `.env.example` should use `DATABASE_URL` / `REDIS_URL`
  - old SQLite env names should be removed or marked legacy

### Phase 1 — Pooled Trial/Lite Runtime

Goal: support Free Trial and Lite at low cost.

- pooled runtime
- pooled data store with `enterprise_id`
- strict quotas
- short retention for trial
- no ERP credentials
- no company-specific connector

### Phase 2 — Dedicated Customer Runtime For Pro

Goal: first serious paid customer can connect systems safely.

- `customer-template` image
- one runtime per Pro company
- runtime mounts enabled agent modules
- per-company secrets
- dedicated schema or DB
- ERP/DingTalk/Excel connectors
- boss daily report / risk alert jobs

### Phase 3 — Automation

Trigger: multiple Pro customers or repeated manual deploy work.

- `ops/add-customer` provisions runtime, env, storage, data plane, tenant rows
- `ops/promote-lite-to-pro` exports/imports data and updates registry
- runtime health and version reporting
- standardized rollback window

### Phase 4 — K8s / Cloud Orchestration

Trigger: 20+ customers, Railway cost/limits, or need for finer isolation.

- platform-web horizontally scalable
- platform-worker separate from web
- customer runtime as deployment/namespace
- secrets manager
- object storage
- queue-based workers
- per-customer resource limits

K8s should be an infrastructure migration, not a product architecture rewrite.

## 9. Why Not One Multi-Tenant Harness For Everything

Pure multi-tenant harness is attractive for cost, but unsafe as the universal
model:

- every query depends on correct `enterprise_id`
- cache keys must always be namespaced
- queue payloads must never lose tenant context
- LLM tools can be called with wrong context unless tool layer is strict
- ERP credentials and private business data raise blast radius
- customer-specific integrations become conditional spaghetti

However, pooled runtime is still appropriate for Free Trial and Lite because:

- no ERP connection
- lower data sensitivity
- strict quotas and retention
- low price point requires shared infrastructure

The design decision is therefore not "pooled vs dedicated"; it is:

```text
pooled for low-risk acquisition tiers,
dedicated/private for data-connected revenue tiers.
```

## 10. Open Decisions

- Lite data store: shared tables with RLS vs schema-per-Lite-tenant after a
  usage threshold.
- Pro default: dedicated schema vs dedicated database. Current recommendation:
  dedicated schema by default, dedicated DB for high-risk or Max.
- Object storage provider: R2 / S3 / Supabase Storage.
- Policy engine implementation: custom Python policy layer first vs OPA/Cedar
  later.
- Embeddings isolation: shared vector table with `enterprise_id` vs per-schema
  vector collections.
- Trial purge policy: 7, 14, or 30 days.

---

## 11. Decisions

| Decision | Rationale |
|---|---|
| Day 1 final boundaries, not final infrastructure | avoids future architecture rewrite while keeping early ops simple |
| Platform is shared control plane | auth, plan, routing, ACL, audit are shared product concerns |
| Runtime is plan-based | Free/Lite need low cost; Pro/Max need isolation and custom connectors |
| Data plane is plan-based | customer data sensitivity rises sharply once ERP/DingTalk/Excel connect |
| Agent is module, not isolation boundary | same company agents should share data and secrets safely |
| Local app is connector, not default runtime | local installs are operationally expensive but useful for private systems |
| HMAC gateway remains the platform/runtime contract | stable migration seam across Railway, K8s, private deployments |
| All tools enforce policy | LLM must not bypass enterprise or intra-enterprise authorization |
