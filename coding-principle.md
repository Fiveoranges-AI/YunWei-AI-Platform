# AI Agent 代码开发原则

在 codebase 规模和复杂度不断增长的过程中，AI agent 容易陷入"打补丁式开发"——每次只关注当前任务能否跑通，累积导致代码腐烂。本文档定义了 agent 在开发和维护代码时应遵循的核心原则，目标是**让代码库在功能持续叠加的同时，保持简洁、可扩展、易修改**。

---

## 一、好代码的核心标准 (What)

Agent 编写的代码应满足以下五项标准：

1. **可读性 (Readable)**：命名清晰、结构统一，让下一个读者（很可能是另一个 agent）在零项目上下文的情况下也能看懂意图。
2. **可维护性 (Maintainable)**：模块化清晰、边界明确，修改一处不应引发其他无关功能的回归。
3. **可扩展性 (Extendable)**：新增功能应通过添加新代码完成，而非大规模改写现有代码。
4. **健壮性 (Robust)**：错误显式抛出、失败模式可观察，不静默吞掉异常，不掩盖问题。
5. **简洁性 (Simple)**：追求逻辑的直接和朴素，拒绝"炫技"写法。冗长但清晰，胜过简短但晦涩。

---

## 二、核心开发原则 (How)

### 1. 高内聚，低耦合 (High Cohesion, Low Coupling)
相关的逻辑聚拢在一起，一个函数/模块只做一件事；不相关的逻辑严格隔离。判断标准：**修改 A 模块时，需要打开几个其他模块才能确信不会破坏东西？答案应该是"很少"。**

### 2. KISS 原则 (Keep It Simple)
优先选择直白、命令式、流程清晰的写法。避免嵌套超过 3 层、单函数超过一屏、一行链式调用塞 5 个操作。**朴素 > 巧妙。**

### 3. 审慎 DRY (Cautious DRY)
避免重复，但**警惕过早抽象**。看到重复时先问：这两处是同一个概念，还是只是长得像？只有在 3 处以上、语义确实相同、且会一起变化时才抽象。**错误的抽象比重复更昂贵。**

### 4. 显式优于隐式 (Explicit over Implicit)
依赖通过参数显式传入，不依赖全局状态、隐式上下文、单例。错误明确抛出，不返回 `null` 掩盖。函数签名应自包含——光看签名就能知道它需要什么、会产生什么。

### 5. 有意义的命名 (Meaningful Naming)
变量、函数名应携带**意图和单位**：`timeoutMs` 而非 `t`，`retryCountRemaining` 而非 `count`。注释只解释 **why**（为什么这么做、有什么坑），不解释 **what**（代码已经说了 what）。

### 6. 可验证性 (Verifiability)
每个改动都必须能通过快速、确定的反馈回路验证：类型检查、linter、相关测试。**没有反馈回路的开发等于盲飞**。新增功能必须配套测试；项目缺少测试基础设施时，先补建再继续。

## 三、抵抗复杂度增长的纪律 (Discipline)

随着 codebase 增长，以下纪律比单条原则更重要：

### 1. 就地修复，不要叠加补丁
发现现有代码不适合新需求时，**重构它**，而不是在外面套一层 if/else 绕过。如果当下无法重构，至少用 TODO 标注，不要假装看不见。

### 2. 出现复杂度警报时停下来
以下信号意味着**该重构，不该继续加功能**：
- 函数超过一屏、嵌套超过 3 层
- 单个文件什么都做（典型如 `utils.ts` 越来越大）
- 修改一处需要同步改 5 处以上
- 出现"我不敢动这块代码"的区域
- 同一逻辑在 3 处以上重复且确实是同一概念

### 3. 小步提交，可逆为先
每个 commit 完成一个原子的、可测试的改动。不要混合重构和功能改动。破坏性操作（删字段、改 schema）要特别谨慎，优先 deprecate 而非直接删除。

### 4. 记录决策
非显然的设计决策需要写下来——注释、ADR、或 `AGENTS.md`。**你今天的决策，明天的 agent 看不到，除非你写下来。**

---

## 四、决策时的优先级 (Trade-offs)

当原则冲突时，按以下顺序权衡：

| 优先 | 让步于 |
|------|--------|
| 正确性 | 一切 |
| 可验证 | 简洁 |
| 可理解 | 性能 |
| 显式 | 隐式 |
| 重复 | 错误的抽象 |
| 删代码 | 加代码 |
| 重构 | 打补丁 |

---

## 五、总结

> **判断标准一句话**：写代码时假设——下一个修改它的 agent 只能看到这个文件，没有项目历史，且只有一次机会改对。在这个假设下你还敢交付，才算合格。

好代码不是教条遵守清单的结果，而是**在具体上下文中做出恰当权衡**的结果。这份文档提供的是默认偏好，不是绝对规则。当你不确定时——**保守选择，显式标注，小步验证**。
